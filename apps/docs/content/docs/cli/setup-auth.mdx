## init

```bash
vars init
```

Sets up a new vars project:

1. Creates `config.vars`, the encrypted secrets file (safe to commit)
2. Creates `.varskey`, the master encryption key (gitignored)
3. Updates `.gitignore` to exclude key and swap files
4. Adds `#vars` import to `package.json` scripts

If a `.env` file exists, vars offers to import it. It reads the key-value pairs, guesses Zod schemas from the values, and encrypts everything before writing.

If `config.vars` already exists but `.varskey` is missing (e.g., a fresh clone), running `init` again detects the incomplete setup and creates the missing key file.

**Flags**

| Flag | Short | Description |
|------|-------|-------------|
| `--file <path>` | `-f` | Path to the `.env` file to import (default: `.env`) |
| `--env <name>` | | Environment name for imported values (default: `dev`) |

```bash
# Import from a non-default path, label values as "staging"
vars init --file .env.staging --env staging
```

## Key Management

Subcommands for managing your encryption key.

<Accordions type="single">
<Accordion title="key init — generate a key file">

```bash
vars key init
```

Generates a new `.varskey` file. Run this on a fresh clone that already has a `config.vars` committed.

</Accordion>
<Accordion title="key export — export key for CI">

```bash
vars key export
```

Prints the base64-encoded master key to stdout. Requires the master PIN; owner PINs cannot export. Use this to set the `VARS_KEY` environment variable in CI where there's no interactive PIN prompt.

```bash
# Copy key to clipboard, then paste into your CI secrets
vars key export | pbcopy
```

<Callout type="warn">This outputs your raw encryption key. Only use it to set CI secrets. Never share it over Slack, email, or commit it to git.</Callout>

</Accordion>
<Accordion title="key rotate — re-encrypt with a new key">

```bash
vars key rotate
```

Generates a new encryption key, re-encrypts values in every `*.vars` file, and replaces `.varskey`. The old key is overwritten.

Before rotating, export your current key as a backup:

```bash
vars key export > vars-key.bak
vars key rotate
```

After rotating, re-export the key to update your CI secrets.

</Accordion>
</Accordions>

## Multi-PIN

Owner-scoped PINs for team access control. See [Multi-PIN](/docs/security/multi-pin) for the full explanation.

<Accordions type="single">
<Accordion title="pin create — create a scoped PIN">

```bash
vars pin create <owner>
```

Creates a scoped PIN for an owner. Requires the master PIN. The CLI derives a sub-key from the master key, wraps it with the new PIN, and re-encrypts fields tagged with that owner.

```bash
# Create a PIN for the backend team
vars pin create backend-team

# The backend team can now use their PIN with show/hide
vars show   # enter backend-team PIN → only their fields decrypt
```

Owner names may only contain letters, digits, hyphens, and underscores.

</Accordion>
<Accordion title="Revoking an owner PIN">

There is no `vars pin delete` command. To revoke an owner's access, remove their `owner=<name>` line from `.varskey` and redistribute the key file. The key file is plain text, so any editor works.

</Accordion>
</Accordions>

## Diagnostics

<Accordions type="single">
<Accordion title="doctor — run health checks">

```bash
vars doctor
```

Runs health checks and prints a summary. Exits 1 if any check fails.

**Files:** `config.vars` exists, `.varskey` exists, codegen is up to date.

**Security:** `.varskey` is gitignored, pre-commit hook installed, all values encrypted.

**Secrets Health:** No expiring or deprecated secrets (warns if `@expires` is within 30 days).

```text
◆  Files
│  ✓ config.vars found
│  ✓ .varskey found
│  △ codegen up to date
│      Run vars gen to regenerate

◆  Security
│  ✓ .varskey is gitignored
│  △ Pre-commit hook installed
│      Run vars init to install

◆  Secrets Health
│  ✓ No expiring or deprecated secrets
```

Run it in CI to catch drift before it bites you in prod.

</Accordion>
<Accordion title="completions — shell autocomplete">

```bash
vars completions <shell>
```

Writes a shell completion script to stdout. Redirect it to wherever your shell loads completions from.

| Shell | Setup |
|-------|-------|
| `bash` | `vars completions bash >> ~/.bash_completion` |
| `zsh` | `vars completions zsh > ~/.zsh/completions/_vars` |
| `fish` | `vars completions fish > ~/.config/fish/completions/vars.fish` |

Open a new shell after running and tab-completion works for all vars subcommands.

</Accordion>
</Accordions>
