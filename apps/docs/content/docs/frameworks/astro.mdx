## Setup

Wrap your dev and build scripts with `vars run`:

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- astro dev",
    "build": "vars run --env prod -- astro build"
  }
}
```

`vars run` decrypts `.vars/vault.vars`, validates against your Zod schemas, and sets `process.env` before Astro starts. No adapter package needed.

## Access variables

Server-side (`.astro` components, API routes, SSR):

```ts
const dbUrl = process.env.DATABASE_URL;
```

Client-side: prefix the variable name with `PUBLIC_`:

```vars
// .vars/unlocked.vars
PUBLIC_API_URL  z.string().url()
  @dev  = http://localhost:3000
  @prod = https://api.example.com
```

Then access it in any component, including client scripts:

```ts
const apiUrl = import.meta.env.PUBLIC_API_URL;
```

Astro uses `PUBLIC_*`, not `VITE_*`. Because `vars run` sets `process.env` before Astro starts, `PUBLIC_*` vars are picked up by Astro automatically.

## CI / CD

Set `VARS_KEY` as a secret in your CI environment:

```bash
VARS_KEY=<your-master-key> vars run --env prod -- astro build
```

## Tips

- Only `PUBLIC_*` vars reach the client. Everything else is server-only.
- Auto-regenerates `env.generated.ts` when the vault changes. No need to run `vars gen` by hand.
- Schema errors throw before the build finishes. Bad values don't make it to production.
- In local development, `vars run` prompts for your PIN to decrypt the key file.
