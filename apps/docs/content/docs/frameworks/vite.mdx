description: Encrypted, typed env vars in Vite. Also works with SvelteKit, Nuxt, and Remix.

## Setup

Wrap your dev and build scripts with `vars run`:

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- vite",
    "build": "vars run --env prod -- vite build"
  }
}
```

`vars run` decrypts `.vars/vault.vars`, validates against your Zod schemas, and sets `process.env` before Vite starts. No adapter package needed.

## Access variables

Server-side, read from `process.env`:

```ts
// src/server/db.ts
const dbUrl = process.env.DATABASE_URL;
```

For client-side, prefix with `VITE_`:

```vars
VITE_API_URL  z.string().url()
  @dev  = http://localhost:3000
  @prod = https://api.yourapp.com
```

Then access it through `import.meta.env`:

```ts
// src/lib/api.ts
const apiUrl = import.meta.env.VITE_API_URL;
```

Because `vars run` sets `process.env` before Vite starts, `VITE_*` vars are picked up by Vite's native env handling and land in `import.meta.env` for client-side use.

## Typed accessors

Run `npx vars gen` once to get `env.generated.ts`. It auto-regenerates when your `.vars` file changes:

```ts
import { env } from "./env.generated";

const apiUrl = env.VITE_API_URL; // string, not string | undefined
```

## SvelteKit, Nuxt, Remix

`vars run` works with any Vite-based framework. Just wrap the framework's CLI command:

```json title="SvelteKit — package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- vite dev",
    "build": "vars run --env prod -- vite build"
  }
}
```

```json title="Nuxt — package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- nuxt dev",
    "build": "vars run --env prod -- nuxt build"
  }
}
```

`VITE_*` for client-side, everything else server-only. Works the same across all Vite-based frameworks.

## CI / CD

Set `VARS_KEY` as a secret in your CI environment:

```bash
VARS_KEY=<your-master-key> vars run --env prod -- vite build
```

## Tips

- Only `VITE_*` vars land in `import.meta.env`. Everything else stays server-side — no accidental leaks if you forget the prefix.
- Schema errors throw before the build finishes. Bad values don't make it to production.
- In local development, `vars run` prompts for your PIN to decrypt the key file.
