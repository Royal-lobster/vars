## Setup

Wrap your dev and build scripts with `vars run`:

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- next dev",
    "build": "vars run --env prod -- next build"
  }
}
```

`vars run` decrypts `.vars/vault.vars`, validates your schema, and injects `process.env` before your framework starts. No adapter packages, no plugins — it works with anything that reads `process.env`.

If you ran `vars init` in a project with a known framework, the dev script was wrapped automatically.

## Generating typed exports

Run `vars gen` once to produce a typed module. The file re-generates whenever your `config.vars` changes.

```ts
import { vars, clientVars } from "#vars";

// vars: all variables — server-only access
const db = vars.DATABASE_URL; // string (validated)

// clientVars: only @public variables — safe to pass to the browser
const apiUrl = clientVars.NEXT_PUBLIC_API_URL;
```

`vars` contains every variable in your schema. `clientVars` contains only variables marked `@public`, which maps directly to each framework's browser-safe prefix convention.

`process.env` still works if you prefer — vars injects everything there too. The `#vars` import adds type safety and removes `undefined` from every value that passed validation.

## Client-side variables

Mark a variable `@public` in `config.vars` to include it in `clientVars` and skip encryption in `vault.vars`:

```vars title="config.vars"
DATABASE_URL  z.string().url()
  @dev  = postgres://localhost:5432/myapp_dev
  @prod = postgres://prod-db.internal:5432/myapp

NEXT_PUBLIC_API_URL  z.string().url()
  @public
  @dev  = http://localhost:3000
  @prod = https://api.myapp.com

NEXT_PUBLIC_POSTHOG_KEY  z.string()
  @public
  @dev  = phc_dev_000000000000000000000000000000
  @prod = phc_live_abc123def456
```

`@public` variables are stored unencrypted in `vault.vars` and appear in `clientVars`. Everything else is encrypted and server-only.

Each framework uses its own prefix for browser-exposed variables:

| Framework | Prefix | Client access |
|-----------|--------|---------------|
| Next.js | `NEXT_PUBLIC_*` | `process.env.NEXT_PUBLIC_X` or `clientVars.NEXT_PUBLIC_X` |
| Vite / SvelteKit | `VITE_*` | `import.meta.env.VITE_X` |
| Astro | `PUBLIC_*` | `import.meta.env.PUBLIC_X` |
| Nuxt | `NUXT_PUBLIC_*` | `useRuntimeConfig().public.x` |

Because `vars run` sets `process.env` before the framework starts, prefixed variables are picked up automatically. Server-only frameworks (Express, Fastify, NestJS) don't need prefixes — all variables are available server-side.

## Platform targets

For edge and serverless runtimes that don't use `process.env`, generate a platform-specific output:

```bash
# Cloudflare Workers — emits a wrangler-compatible vars binding
vars gen config.vars --platform cloudflare

# Vercel Edge — emits edge runtime compatible output
vars gen config.vars --platform vercel-edge
```

The generated module adjusts its runtime bindings to match the target platform while keeping the same `import { vars } from "#vars"` interface in your code.

## Framework examples

### Next.js

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- next dev",
    "build": "vars run --env prod -- next build",
    "start": "vars run --env prod -- next start"
  }
}
```

```vars title="config.vars"
DATABASE_URL  z.string().url()
  @dev  = postgres://localhost:5432/myapp_dev
  @prod = postgres://prod-db.internal:5432/myapp

NEXTAUTH_SECRET  z.string().min(32)
  @dev  = dev-secret-at-least-32-characters-long
  @prod = enc:v1:aes256gcm:...

NEXT_PUBLIC_API_URL  z.string().url()
  @public
  @dev  = http://localhost:3000
  @prod = https://api.myapp.com
```

```ts title="app/lib/db.ts"
import { vars } from "#vars";

export const db = createPool(vars.DATABASE_URL);
```

### Vite

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- vite",
    "build": "vars run --env prod -- vite build"
  }
}
```

```vars title="config.vars"
VITE_API_URL  z.string().url()
  @public
  @dev  = http://localhost:8080
  @prod = https://api.myapp.com

VITE_STRIPE_PUBLISHABLE_KEY  z.string()
  @public
  @dev  = pk_test_abc123
  @prod = pk_live_def456
```

Client-side access uses Vite's standard `import.meta.env`:

```ts
const apiUrl = import.meta.env.VITE_API_URL;
```

Or use `clientVars` for type safety without `undefined`:

```ts
import { clientVars } from "#vars";

const apiUrl = clientVars.VITE_API_URL; // string, never undefined
```

### Astro

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- astro dev",
    "build": "vars run --env prod -- astro build"
  }
}
```

```vars title="config.vars"
PUBLIC_SITE_URL  z.string().url()
  @public
  @dev  = http://localhost:4321
  @prod = https://myapp.com

CMS_API_KEY  z.string().min(16)
  @dev  = cms-dev-key
  @prod = enc:v1:aes256gcm:...
```

Server-side Astro components and API routes use `vars.CMS_API_KEY`. Client-side components use `import.meta.env.PUBLIC_SITE_URL` or `clientVars.PUBLIC_SITE_URL`.

### Remix

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- remix vite:dev",
    "build": "vars run --env prod -- remix vite:build"
  }
}
```

```vars title="config.vars"
SESSION_SECRET  z.string().min(32)
  @dev  = dev-session-secret-min-32-characters
  @prod = enc:v1:aes256gcm:...

DATABASE_URL  z.string().url()
  @dev  = postgres://localhost:5432/myapp_dev
  @prod = postgres://prod-db.internal:5432/myapp
```

In loaders and actions, import from `#vars` for typed access. Remix's server/client boundary ensures `vars` is never bundled into the browser.

### Express / Node

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- tsx watch src/index.ts",
    "start": "vars run --env prod -- node dist/index.js"
  }
}
```

```vars title="config.vars"
PORT  z.coerce.number().int().min(1).max(65535)
  @default = 3000

DATABASE_URL  z.string().url()
  @dev  = postgres://localhost:5432/myapp_dev
  @prod = postgres://prod-db.internal:5432/myapp

JWT_SECRET  z.string().min(32)
  @dev  = dev-jwt-secret-at-least-32-characters
  @prod = enc:v1:aes256gcm:...
```

```ts title="src/index.ts"
import { vars } from "#vars";

const app = express();
app.listen(vars.PORT);
```

No prefixes needed — all variables are server-side.

## CI / CD

Set `VARS_KEY` as a secret in your CI environment:

```bash
VARS_KEY=<your-master-key> vars run --env prod -- npm run build
```

No PIN prompt in CI — `VARS_KEY` provides the decryption key directly.

## Tips

- Schema failures throw at startup. A bad URL or missing required variable breaks before anything ships.
- Only `@public` variables reach the browser. Everything else is server-only, even if you forget to think about it.
- In local development, `vars run` prompts for your PIN to decrypt the vault.
- The `clientVars` export is typed to only include `@public` variables — passing it to a client bundle is safe by construction.
