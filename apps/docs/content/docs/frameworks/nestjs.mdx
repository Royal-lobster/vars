## Setup

Wrap your dev and build scripts with `vars run`:

```json title="package.json"
{
  "scripts": {
    "dev": "vars run --env dev -- nest start --watch",
    "build": "vars run --env prod -- nest build",
    "start:prod": "vars run --env prod -- node dist/main"
  }
}
```

`vars run` decrypts `.vars/vault.vars`, validates against your Zod schemas, and sets `process.env` before NestJS starts. No adapter package needed.

## Access variables

Read directly from `process.env`:

```ts title="app.service.ts"
import { Injectable } from "@nestjs/common";

@Injectable()
export class AppService {
  getDatabaseUrl(): string {
    return process.env.DATABASE_URL!;
  }
}
```

Or use the generated typed accessors for full type safety:

```ts
import { env } from "./env.generated";

const dbUrl = env.DATABASE_URL; // string, not string | undefined
```

Run `npx vars gen` to generate `env.generated.ts`. It auto-regenerates when your `.vars` file changes.

## CI / CD

Set `VARS_KEY` as a secret in your CI environment:

```bash
VARS_KEY=<your-master-key> vars run --env prod -- nest build
```

## Tips

- NestJS is server-only, so there's no client/server variable prefix split to worry about.
- If a schema validation fails, `vars run` throws before NestJS boots. Your app won't start with bad vars.
- In local development, `vars run` prompts for your PIN to decrypt the key file.
